# RULES: Правила работы с файлами плана

## Страховка от перезаписи

Эти правила ОБЯЗАТЕЛЬНЫ для каждого чата, работающего над фичей.

### Общий принцип: точечные изменения, не полная перезапись

1. **Читать файлы целиком перед изменением** — всегда через `get_file_contents`, не по памяти
2. **Изменять только отведённые секции** — если задача касается `retrieval.py`, трогать только функции `retrieve()` и `rerank_hits()`, не переписывать весь файл
3. **Проверять перед коммитом** — прочитать изменённый файл после записи, убедиться что остальной код не тронут
4. **Коммитить пофайлово** — один файл = один коммит с понятным сообщением

### Правила обновления PROGRESS.md

- Обновлять **только строку своего шага** в таблице
- Добавлять запись в секцию `## Notes` — не редактировать чужие записи
- Если шаг не завершён — статус `[~]`, если проблема — `[!]` с описанием в Notes

### Признаки того что что-то пошло не так

Если при работе возникают следующие ситуации — **остановиться и сообщить пользователю**:
- Файл, который нужно изменить, содержит неожиданный код (не совпадает с описанием в CONCEPT.md)
- Миграция БД требует изменений в существующих таблицах (не только создание новых)
- API-эндпоинт с таким именем уже существует
- В `retrieval.py` уже есть вызов reranker (значит кто-то уже начал реализацию)

### Что НЕЛЬЗЯ делать

- Переписывать существующие методы в `settings_service.py` — только добавлять новые
- Менять сигнатуры функций `retrieve()` и `retrieve_multi_vault()` — только добавлять вызов внутри
- Удалять ключи `reranker.*` из `platform_settings` до Шага 8
- Изменять `CONCEPT.md` — этот файл фиксирован, не редактируется в процессе реализации
