# Промпт для ИИ-ассистента — Campaign Update Mode

Этот файл описывает как вести работу по плану реализации Campaign Update Mode.
Используй его как системную инструкцию в начале каждого рабочего чата.

---

## Инструкция для ИИ-ассистента

Ты реализуешь фичу **Campaign Update Mode** для платформы Mercer.
Mercer — full-stack RAG/LLM платформа (Python/FastAPI backend, vanilla JS frontend,
PostgreSQL, LanceDB, Redis). Репозиторий: `https://github.com/Annener/mercer`.

### Шаг 1 — Прочитай контекст проекта

Перед началом работы ОБЯЗАТЕЛЬНО прочитай следующие файлы из репозитория:

1. `context/architecture.md` — общая архитектура сервисов
2. `context/rag-backend-services.md` — сервисы rag-backend, включая full_document_service
3. `context/db_schema.md` — схема PostgreSQL
4. `context/shared_contracts.md` — Pydantic-схемы
5. `context/api_routes.md` — существующие API-роутеры
6. `context/frontend.md` — структура фронтенда

### Шаг 2 — Прочитай концепт

Прочитай `plan-update/concept.md` — это компактная справочная копия концепта.

### Шаг 3 — Прочитай статус плана

Открой `plan-update/status.md`. Найди **первую фазу со статусом `[ ]` не начата
или `[~]` в процессе** — это твоя активная фаза.

### Шаг 4 — Прочитай файл активной фазы

Открой соответствующий файл фазы:
- Фаза 1: `plan-update/phase-1-git-infrastructure.md`
- Фаза 2: `plan-update/phase-2-data-model.md`
- Фаза 3: `plan-update/phase-3-executor.md`
- Фаза 4: `plan-update/phase-4-api.md`
- Фаза 5: `plan-update/phase-5-sse-frontend.md`

Прочитай весь файл фазы: цели, задачи, требования к реализации, тесты,
критерии готовности.

Дополнительно прочитай файлы кода, указанные в разделе
"Контекст для чтения" файла фазы.

### Шаг 5 — Начни работу

Выполняй задачи фазы по порядку. Для каждой задачи:
1. Прочитай релевантный существующий код перед написанием нового
2. Следуй паттернам, уже используемым в проекте (async/await, ORM-стиль,
   naming convention, структура тестов)
3. Пиши тесты вместе с кодом, не откладывай на потом
4. При отступлении от плана — зафиксируй причину в `plan-update/status.md`

### Шаг 6 — Обнови статус после завершения

После завершения всех задач фазы:

1. Открой `plan-update/status.md`
2. Отметь все выполненные чекбоксы: `[ ]` → `[x]`
3. Обнови статус фазы в таблице: `[ ] не начата` → `[x] завершена`
4. Заполни строку "Дата завершения" и "Итог фазы" (2-3 предложения что было сделано)
5. Обнови строку "Последнее обновление" внизу файла
6. Если были нестандартные решения — добавь в раздел "Известные проблемы / решения"
7. Сделай коммит изменений в main (включая status.md)

## Правила работы с кодом

**Читай перед тем как писать**: всегда смотри существующий код прежде чем
создавать новый файл. Особенно для сервисов — изучи соседний похожий сервис.

**Паттерны проекта**:
- Async everywhere: `async def`, `AsyncSession`, `asyncio.create_subprocess_exec`
- ORM: SQLAlchemy 2.x mapped_column стиль (см. models.py)
- Тесты: pytest + pytest-asyncio, моки через `unittest.mock`
- Логирование: `logging.getLogger(__name__)`
- Ошибки: кастомные Exception-классы, не голые Exception
- Alembic: migration файлы с upgrade/downgrade

**Не трогать без необходимости**:
- Существующие миграции (только добавлять новые)
- Существующие shared_contracts (только добавлять новые модели)
- Механизм индексации (watchdog, indexer) — update-mode не вводит свой reindex

**Тесты**:
- Новые тесты кладутся в `rag-backend/app/tests/` по именованию `test_update_mode_*.py`
- Используй `tmp_path` pytest-фикстуру для файловых операций в тестах
- Моки для Redis: `AsyncMock` с нужными return_value
- Моки для LLM: возвращать корректный JSON строкой

## Что делать если что-то непонятно

1. Сначала поищи ответ в файлах контекста (`context/*.md`)
2. Если не нашёл — прочитай соответствующий исходный код в репозитории
3. Если противоречие между планом и реальным кодом — следуй реальному коду,
   зафикируй расхождение в `plan-update/status.md`
4. Не придумывай архитектуру которой нет в проекте — используй существующие паттерны
