# Spec-05c: Settings UI — Tabs 4-7 (Embedding Models, Parameters, Pipelines, Worlds)

Перед выполнением прочитай `Spec-05a` и `Spec-05b`. Вкладки «Домены», «Vault'ы», «Генеративные модели» уже реализованы.

**Зависит от:** `Spec-05a`, `Spec-05b`.

**Цель:** Реализовать оставшиеся вкладки страницы настроек: «Embedding-модели», «Параметры», «Pipelines», «Миры».

## Контекст

**Прочитать перед реализацией:**
- `rag-backend/static/js/settings.js` — расширить существующий класс
- `rag-backend/static/css/chat.css` — добавить стили для новых элементов

## Задачи

### 1. Реализовать рендеринг вкладки «Embedding-модели»

Метод `renderEmbeddingModelsTab()`.

**Требования:**
- Список моделей из `api.getEmbeddingModels()`.
- Карточка: `display_name` (или `model_id`), `provider`, `dimensions`, список привязанных vault'ов (текстом, получить из `api.getSettingsVaults()` и отфильтровать по `embedding_model_id`).
- Кнопки: `Проверить доступность` (вызов `api.checkEmbeddingModel(id)`, показать ✅ + latency + dimensions или ❌ + ошибку), `Редактировать`, `Удалить`.
- Кнопка «+ Новая модель».

**Модалка создания/редактирования:**
- Поля: `model_id` (при создании), `provider` (select: `ollama`, `openai_compatible`), `display_name` (опционально), `model_name` (строка), `base_url` (валидный URL), `api_key` (опционально, password input), `dimensions` (int, >0), `timeout_seconds` (int, дефолт 30), `max_retries` (int, дефолт 3).
- При `provider = ollama` поле `api_key` скрыть или задизейблить (Ollama обычно без ключа).

**Проверка доступности:**
- Вызвать `api.checkEmbeddingModel(id)`. Результат: `ok`, `latency_ms`, `dimensions`, `error`.
- Показать в модалке или тосте.

**Удаление:**
- Если модель привязана к одному или нескольким vault'ам (поле `connected_vaults.length > 0`), кнопка `disabled` с тултипом «Невозможно удалить: привязано N vault'ов».
- Иначе — подтверждение и вызов `api.deleteEmbeddingModel(id)`.

### 2. Реализовать рендеринг вкладки «Параметры»

Метод `renderParamsTab()`.

**Требования:**
- Получить все параметры через `api.getSettingsParams()`.
- Сгруппировать по `group_name` (группы из БД: `retrieval`, `chunking`, `chat`, `reranker`, `sidecar`).
- Для каждого параметра: `label`, input (тип зависит от `value_type`), кнопка «↺ Дефолт» (сброс к default из `DEFAULTS`), иконка `?` с тултипом (содержимое — `hint` из БД).
- Кнопка «Сбросить все к дефолтам» вверху.

**Типы полей ввода:**
- `bool` → `<input type="checkbox">`
- `int` или `float` → `<input type="number" step="any">`
- `str` → `<input type="text">`

**Логика сохранения:**
- По `change` или `blur` (для текстовых полей — по `blur`, для чекбоксов — по `change`).
- Вызвать `api.updateSettingsParam(key, value)`. При успехе — обновить локальное состояние, при ошибке `422` — показать тост «Недопустимое значение для типа ...» и восстановить предыдущее значение.
- При клике на «↺ Дефолт» — установить значение из дефолтов (хардкод в `settings.js` на основе `DEFAULTS` из Spec-02a или получить отдельным API? Лучше захардкодить в JS те же значения, что и в бэкенде, чтобы не делать лишний запрос).

**Обновление статусной плашки:**
- После изменения `pdf_sidecar.url` или создания/удаления vault — вызвать `settingsManager.updateStatusBanner()`.

### 3. Реализовать рендеринг вкладки «Pipelines»

Метод `renderPipelinesTab()`.

**Требования:**
- Фильтр по домену (select с доменами, исключая `default`).
- При выборе домена — загрузить пайплайны через `api.getPipelines(domainId)`.
- Список карточек: `name`, `version`, `steps` (количество шагов), badge `активен`/`архив`.
- Кнопки: `Редактировать` (открывает JSON-редактор шагов), `Дублировать` (открывает форму создания с предзаполненными полями), `Деактивировать` (вызов `api.deletePipeline(id)` — soft delete), `Активировать` (если не активен — кнопка `api.activatePipeline(id)`).
- Кнопка «+ Новый pipeline».

**Модалка создания/редактирования:**
- При создании: `pipeline_id` (slug), `domain_id` (select), `name`, `description` (textarea), `steps` (JSON-редактор — можно использовать `<textarea>` для ввода JSON с валидацией), `final_composition` (JSON), чекбокс `is_active`.
- При редактировании: все поля, кроме `pipeline_id` (не редактируется). Сохранение через `api.updatePipeline(id, data)` создаёт новую версию.
- Валидация JSON на клиенте перед отправкой: проверить соответствие схеме (наличие обязательных полей, типы). При ошибке показать сообщение.

**Дублирование:**
- Загрузить существующий пайплайн, открыть модалку создания с предзаполненными полями, но с новым `pipeline_id` (можно предложить сгенерировать на основе старого + `_copy`).

### 4. Реализовать рендеринг вкладки «Миры»

Метод `renderWorldsTab()`.

**Требования:**
- Фильтр по vault'у (select из списка vault'ов, полученных через `api.getSettingsVaults()`).
- При выборе vault — загрузить миры через `api.getWorlds(vaultId)`.
- Для каждого мира: карточка с `name`, `world_id`, `path_prefix`, `is_active` (toggle), кнопка `Редактировать`. При раскрытии — список кампаний мира (получить через `api.getWorldCampaigns(worldId)`).
- Кнопка «+ Новый мир».

**Список кампаний:**
- Каждая кампания: `name`, `campaign_id`, `path_prefix`, `is_active` (toggle), кнопка `Редактировать`.
- Кнопка «+ Новая кампания» для каждого мира.
- **Важно:** Нет кнопок «Удалить» для миров и кампаний. Вместо этого отображать текст: «Удаление миров и кампаний выполняется в файловой системе. Удалите файлы и соответствующие документы через окно "Управление хранилищем".»

**Модалка создания/редактирования мира:**
- Поля: `world_id` (slug, только при создании), `name`, `description`, `path_prefix` (должен заканчиваться на `/`), `is_active`.
- `vault_id` выбирается из фильтра (или отдельный select).

**Модалка создания/редактирования кампании:**
- Поля: `campaign_id` (slug, только при создании), `name`, `description`, `path_prefix` (должен начинаться с `path_prefix` мира и заканчиваться на `/`), `is_active`.

**Toggle is_active:**
- Для миров: `api.updateWorld(worldId, { is_active: newValue })`.
- Для кампаний: `api.toggleCampaign(worldId, campaignId)` (или `updateCampaign`).

### 5. Обновить `updateStatusBanner()`

Уже реализовано в Spec-05a. Дополнительно: при изменении `pdf_sidecar.url` через вкладку параметров вызывать `updateStatusBanner()`.

Добавить:

- `.settings-param-group` — группировка по `group_name`.
- `.json-editor` — textarea с моноширинным шрифтом.

## Финальный контракт

- Вкладки «Embedding-модели», «Параметры», «Pipelines», «Миры» полностью рабочие.
- Параметры сохраняются с валидацией, сброс к дефолтам работает.
- Пайплайны можно создавать, редактировать (создание новой версии), деактивировать, активировать.
- Миры и кампании управляются без возможности удаления (только toggle и редактирование).
- Статусная плашка обновляется после изменения параметров или vault'ов.

## Критерии приёмки

- [ ] Embedding-модели: проверка доступности показывает `dimensions`, корректно блокируется удаление при привязке к vault'у.
- [ ] Параметры: все 16 параметров отображаются с правильными контролами, сохранение по blur/change работает.
- [ ] Кнопка «Сбросить все к дефолтам» восстанавливает значения из `DEFAULTS`.
- [ ] Pipelines: фильтр по домену работает, создание пайплайна с валидным JSON сохраняется.
- [ ] Редактирование пайплайна создаёт новую версию с инкрементом (`1.0.0` → `1.0.1`).
- [ ] Деактивация пайплайна (soft delete) убирает его из списка активных, но он остаётся в БД.
- [ ] Миры: создание мира с корректным `path_prefix` (заканчивается на `/`) успешно.
- [ ] Кампания: `path_prefix` валидируется на клиенте (начинается с `path_prefix` мира).
- [ ] Нет кнопок «Удалить» для миров и кампаний, есть информационный текст.
- [ ] Статусная плашка обновляется после изменения `pdf_sidecar.url`.
