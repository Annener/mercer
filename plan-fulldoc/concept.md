# Mercer: концепт выборочной отправки полного документа в LLM

## Цель

Добавить в чат Mercer управляемый пользователем режим, в котором в генеративную модель можно отправлять не только найденные чанки, но и полный текст релевантных документов, если пользователь явно выбрал такие документы.

Подход должен:
- не создавать дополнительную фоновую нагрузку на локальные модели;
- не запускать дополнительные preview-запросы во время ввода сообщения;
- сохранять основную RAG-схему как базовый путь;
- давать пользователю точечный контроль над тем, какие документы будут переданы целиком;
- избегать повторной отправки документов, которые уже были полностью загружены в контекст текущего чата.

## Основная идея

Полный документ не должен автоматически подмешиваться в промпт. Вместо этого система сначала выполняет обычный RAG-проход: переформулирование запроса, retrieval, reranking, отбор релевантных чанков. После этого, если в чате включён специальный режим, пользователю предлагается список релевантных документов-источников, собранный по найденным чанкам.

Пользователь вручную отмечает документы, которые нужно отправить в модель полностью. Для отмеченных документов система выгружает все чанки документа, восстанавливает полный текст и добавляет его в prompt context. Для остальных документов в prompt попадают только найденные чанки, как и в обычном RAG-сценарии.

Итоговый контекст может одновременно содержать:
- полный текст выбранных документов;
- релевантные чанки из невыбранных документов;
- пользовательский вопрос и служебные инструкции пайплайна.

## Пользовательский сценарий

### 1. Включение режима

В интерфейсе чата есть переключатель:
> Разрешить отправку полных документов

Этот флаг действует только как разрешение на дополнительный шаг выбора. Сам по себе он ничего автоматически не отправляет.

### 2. Отправка сообщения

Пользователь отправляет сообщение в чат обычным образом. Система выполняет стандартный pipeline: clarification → query rewriting → retrieval → reranking → сбор списка документов-источников.

### 3. Предложение выбрать документы

Если режим включён, система показывает промежуточный UI-этап. В список попадают только документы, из которых пришли найденные чанки. Для каждого документа показывается:
- название файла;
- ориентировочный размер (токены / символы / чанки);
- признак «уже отправлен целиком в этом чате».

### 4. Подтверждение выбора

Пользователь отмечает нужные документы. После подтверждения:
- для отмеченных документов: полный текст из склейки всех чанков;
- для неотмеченных: только найденные чанки;
- prompt собирается гибридно.

### 5. Дальнейшая работа

Документы, уже отправленные целиком, запоминаются в состоянии чата (`sent_full_document_ids`) и помечаются в следующих итерациях.

## Состояние чата

```python
full_document_mode_enabled: bool          # включён ли режим
sent_full_document_ids: list[str]         # уже отправленные целиком в этом чате
# В pipeline_pause_state при паузе:
# step: "full_document_selection"
# candidates: list[DocumentCandidate]
# saved_hits: list[SearchHit]             # результаты retrieval для resume
# query: str
# pipeline_id: str
```

## Источник полного текста

Вариант A (реализуем в MVP): склейка всех чанков документа через `GET /index/document/{document_id}/chunks?vault_id=...`.
Чанки уже отсортированы по `chunk_index` (отдельная колонка в LanceDB).
Склейка: `'\n\n'.join(chunk.text for chunk in chunks)`.

## Сборка prompt

Порядок секций в итоговом prompt:
1. системные инструкции;
2. служебные инструкции пайплайна;
3. полные тексты выбранных документов (с заголовком);
4. релевантные чанки из остальных документов;
5. пользовательский запрос.

Чанки документов, выбранных для полной отправки, **не дублируются** в chunk-секции.

## Ключевые правила

1. **Без preview-запросов** — выбор формируется только после реального запроса пользователя.
2. **Только релевантные документы** — только источники найденных чанков.
3. **Не предлагать повторно** — `sent_full_document_ids` для UX-оптимизации и защиты от дублей.
4. **Показывать размер** — `estimated_tokens` хранится в Document, вычисляется при индексации.
5. **Убирать дубли** — при сборке prompt чанки выбранных документов исключаются.

## Ограничения и риски

- **Переполнение контекста** — пользователь видит суммарный объём токенов перед подтверждением.
- **Ложное ощущение присутствия** — `sent_full_document_ids` это UX-оптимизация, не строгая гарантия нахождения в контексте LLM.
- **Двухфазный поток** — retrieval → выбор → generation. Промежуточное состояние должно быть визуально понятным.

## MVP scope

1. Флаг `full_document_mode_enabled` в чате.
2. Пауза после retrieval/reranking.
3. Список кандидатов на основе источников чанков.
4. Ручной выбор документов.
5. Склейка полного текста из чанков.
6. Исключение чанков выбранных документов из chunk-секции.
7. Хранение `sent_full_document_ids` в состоянии чата.
8. Отображение `estimated_tokens` / `char_count` рядом с документом.
