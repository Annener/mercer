# TD-10 — Двойная фильтрация по document_ids в retrieve()

**Приоритет:** 🟠 Структурный  
**Файл:** `rag-backend/app/services/retrieval.py`, функция `retrieve()`

## Проблема

Фильтрация по `document_id` применяется дважды:

1. Как `filter_expr` передаётся в `_vector_search` (фильтр на стороне LanceDB)
2. Затем Python-фильтрация по `doc_set` уже в Python

Вторая фильтрация выглядит как страховка, но создаёт путаницу:
Лог пишет `post-filter applied` — читатель думает, что LanceDB сработал неправильно.
Также это маскирует реальную производительность: непонятно, сколько документов
реально отфильтровал LanceDB vs Python.

## Анализ перед исправлением

- [ ] Проверить, действительно ли LanceDB с `filter_expr` корректно
  фильтрует по `document_id` во всех провайдерах
- [ ] Есть ли known-issues с LanceDB filter когда `document_ids` большой список?
- [ ] Оценить — post-filter добавлялся из-за реального бага или «на всякий случай»?
- [ ] Посмотреть git log для этого участка кода

## Ожидаемое исправление

**Если LanceDB filter надёжен:**  
Удалить Python-постфильтрацию, убрать вводящий в заблуждение лог `post-filter applied`.

**Если есть сомнения в LanceDB:**  
Оставить Python-фильтрацию, но явно задокументировать почему и добавить
метрику (сколько документов было отфильтровано повторно) для мониторинга.

## Риски

Средние — нужно убедиться в корректности LanceDB filter перед удалением страховки.
