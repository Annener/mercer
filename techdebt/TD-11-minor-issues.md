## 11-A: import re внутри функции

**Файл:** `rag-backend/app/services/retrieval.py`  

Импорт `re` расположен внутри функции `_score_from_response_text` вместо
уровня модуля. Python кэширует импорты, так что это не критический перф-баг,
но нарушает конвенции и усложняет статический анализ.

**Исправление:** перенести `import re` в начало файла.

## 11-B: f-string в logging вместо lazy evaluation

**Файл:** `rag-backend/app/services/planner.py`

```python
# Плохо:
logger.warning(f"LLRAGPlanner decomposition failed: {e}")

# Хорошо:
logger.warning("LLRAGPlanner decomposition failed: %s", e)
```

С f-string строка форматируется всегда, даже если уровень логирования
не позволит записи попасть в лог. С %-форматированием — только при реальной
записи. Для горячих путей это заметно.

**Исправление:** заменить все `logger.*(f"...")` на `logger.*("...", args)`
во всех сервисах.

## 11-C: _VALIDATION_TTL без реальной проверки

**Файл:** `rag-backend/app/services/pipeline_executor.py`

Константа `_VALIDATION_TTL` помечена комментарием «по концепту»,
что означает TTL не проверяется при `resume`. Это потенциальный баг:
старые/невалидные состояния могут быть возобновлены.

**Анализ:** проверить — нужна ли реальная проверка TTL при resume,
или это сознательное решение? Если нужна — реализовать.

---

## 11-D: Дублирование logging_config.py

**Файлы:**
- `rag-backend/app/logging_config.py`
- `rag-indexer/logging_config.py`

Файлы идентичны (или почти идентичны). При изменении формата логов нужно
менять в двух местах.

**Варианты исправления:**
1. Вынести в `shared_contracts/logging_config.py` (если пакет монтируется в indexer)
2. В `rag-indexer` сделать симлинк или Docker COPY из backend
3. Задокументировать как намеренное дублирование с явным комментарием

**Анализ:** проверить, как `shared_contracts` монтируется в `rag-indexer`
(см. `docker-compose.yml`).
