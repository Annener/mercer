# Mercer — целевая концепция и архитектура

Документ описывает **целевую реализацию** Mercer, к которой должен быть приведён код. Это не описание текущего состояния, а спецификация.

## 1. Архитектурные принципы

- Базовая сущность системы — **Domain**. Любые рабочие сущности (Vault, Document, Tag, Campaign, Chat, Pipeline) существуют только внутри конкретного домена.
- **Vault** — это файловое хранилище внутри домена. В одном домене может быть несколько Vault; все они рассматриваются как единое пространство знаний домена.
- Индексация отделена от появления файла в файловой системе. Файл попадает в retrieval и RAG **только после явного запуска индексации**.
- Retrieval и pipeline‑retrieval работают **по всем enabled‑Vault** домена (multi‑vault), а не по одному Vault.

## 2. Сущности

### 2.1 Domain

Domain — верхнеуровневый контейнер. На уровне домена живут:

- vault’ы;
- кампании;
- теги;
- пайплайны;
- чаты;
- доменные промпты и настройки.

Никакой чат, тег, кампания или pipeline не существует вне домена.

### 2.2 Vault

Vault:

- принадлежит одному домену (`vault.domain_id`);
- описывает привязку к файловой системе и настройку эмбеддингов;
- хранит документы (`Document`) этого хранилища.

Vault **не определяет контекст чата** — чат работает по всем Vault домена.

Поле `enabled` на Vault:

- `enabled = true` — Vault участвует в retrieval и pipeline‑retrieval;
- `enabled = false` — Vault игнорируется retrieval-ом, но его записи в БД не обязаны удаляться.

### 2.3 Document

Document:

- принадлежит ровно одному Vault;
- содержит путь к физическому файлу, статус индексации и служебные поля;
- связан с тегами через таблицу `document_labels`.

Поведение:

- Файл в ФС сам по себе не участвует в retrieval.
- При запуске индексации компонент `rag-indexer`:
  - читает файл;
  - делает распознавание/чанкинг/эмбеддинг;
  - пишет чанки в LanceDB;
  - проставляет `status = 'indexed'`, `indexed_at`.
- При удалении документа через UI:
  - удаляется запись `Document` в Postgres;
  - удаляются соответствующие чанки в LanceDB;
  - **физический файл в ФС не удаляется**.

### 2.4 Tag

Tag — доменная сущность, **не привязанная к Vault**.

Свойства:

- `domain_id` — всегда обязателен;
- `campaign_id` — nullable:
  - `NULL` — общий доменный тег;
  - `NOT NULL` — тег кампании.
- один и тот же тег может навешиваться на документы любых Vault своего домена.

Режимы создания:

- из окна управления документами:
  - создаётся общий доменный тег (`campaign_id = NULL`);
- из окна кампании:
  - создаётся тег кампании (`campaign_id = id кампании`).

### 2.5 Campaign

Campaign:

- принадлежит домену (`campaign.domain_id`);
- **не привязана к Vault**;
- логически сужает контекст RAG по тегам кампании;
- может иметь свой `system_prompt`.

Правила:

- кампания работает по всем enabled‑Vault домена;
- в режиме кампании retrieval фильтрует документы только по тегам этой кампании (и, при необходимости, общим тегам, завернутым в пайплайн);
- если `campaign.system_prompt` не пустой — он **заменяет** доменный системный промпт;
- если `campaign.system_prompt` пустой — используется доменный системный промпт.

### 2.6 Chat

Chat:

- принадлежит домену: `domain_id` — **NOT NULL**;
- может иметь `campaign_id` (режим кампании) или `NULL` (общий режим);
- **не имеет** `vault_id` — чат никогда не привязан к одному Vault.

Режимы:

- **режим кампании** — `chat.campaign_id` задан:
  - retrieval ограничен документами, доступными по тегам кампании;
  - pipeline router может выбирать только пайплайны этой кампании;
- **общий режим** — `chat.campaign_id = NULL`:
  - retrieval работает по всем документам домена (без фильтрации по кампании);
  - pipeline router может выбирать только общие пайплайны (без `campaign_id`).

Чаты фильтруются по домену: при переключении домена пользователь видит только чаты этого домена.

### 2.7 Pipeline

Pipeline:

- принадлежит домену (`pipeline.domain_id`);
- может быть:
  - общим (`campaign_id IS NULL`);
  - кампанийным (`campaign_id` ссылается на конкретную кампанию).
- структура шагов хранится в JSONB (`steps + final_step`).

Доступность пайплайнов:

- в **общем режиме чата** (нет кампании) доступны только пайплайны домена с `campaign_id IS NULL`;
- при **выборе кампании** доступны только пайплайны с `campaign_id = chat.campaign_id`.

## 3. Индексация

Поток:

1. Пользователь помещает файл в файловую систему Vault.
2. UI обнаруживает новый файл и показывает его как неиндексированный.
3. Пользователь явным действием запускает индексацию:
   - создаётся/обновляется запись `Document`;
   - `rag-indexer` выполняет распознавание/чанкинг/эмбеддинг;
   - чанки пишутся в LanceDB;
   - `Document.status` меняется на `indexed`.
4. Повторная индексация (`reindex`) должна быть идемпотентной по документу.

Удаление документа из UI:

- удаляется `Document` + связанные `DocumentLabel` + чанки в LanceDB;
- файл в ФС остаётся.

## 4. Работа тегов

- Теги живут на уровне домена.
- Тег кампании может быть назначен на документ из любого Vault этого домена.
- Общие теги используются как сквозные метки для общих материалов (инструкции мастера, генераторы и т.д.).

Два режима:

- **кампанийные теги**:
  - создаются и управляются из окна кампании;
  - используются для RAG в рамках этой кампании;
- **общие теги**:
  - создаются из окна управления документами;
  - используются:
    - напрямую в общем режиме;
    - как источники в конкретных шагах пайплайна (и для кампаний, и для общего режима).

## 5. Контекст чата и retrieval

Чат:

- жёстко привязан к домену;
- режим определяется `campaign_id`.

Retrieval:

- работает по всем **enabled** Vault домена;
- при активной кампании:
  1. вычисляются разрешённые теги: теги кампании + (для пайплайна) явные общие теги шага;
  2. по этим тегам находятся документы во всех enabled‑Vault домена;
  3. retrieval ограничивается этим списком `document_ids`;
  4. если по тегам кампании не найдено ни одного документа — возвращается **пустой результат**, но **не** весь домен.
- в общем режиме:
  - retrieval работает по всему домену;
  - кампанийные теги не участвуют в фильтрации.

Multi‑vault:

- retrieval и pipeline‑retrieval всегда работают через список `vault_ids` домена;
- отдельный вызов по одному Vault не используется как основной путь.

## 6. Pipeline: модель и поведение

### 6.1 Общая идея

Pipeline — это сценарий улучшенного ответа, который:

- разбивает задачу на несколько retrieval‑шагов по разным тегам;
- на каждом шаге формирует отдельный контекст и промежуточный ответ;
- затем финальный LLM‑шаг собирает всё вместе и выдаёт ответ пользователю.

Pipeline не привязан к Vault; он работает с документами всех enabled‑Vault домена.

### 6.2 JSON‑схема пайплайна

Структура в `pipelines.steps` (JSONB) и `final_composition`:

```json
{
  "version": "2",
  "steps": [
    {
      "id": "rules",
      "type": "retrieval",
      "title": "Правила боя",
      "tag_id": "UUID-тега",
      "top_k": 5,
      "prompt": "..."
    },
    {
      "id": "lore",
      "type": "retrieval",
      "title": "Лор региона",
      "tag_id": "UUID-тега",
      "top_k": 8,
      "prompt": "..."
    }
  ],
  "final_step": {
    "type": "final",
    "prompt": "..."
  }
}
```

Правила:

- `steps[]` — только шаги типа `"retrieval"`;
- у каждого retrieval‑шага:
  - ровно один `tag_id`;
  - `top_k` (целое число);
  - обязательный `title` для UI и логов;
  - `id` генерируется builder‑ом (уникален в пределах пайплайна);
- `final_step`:
  - всегда присутствует;
  - имеет тип `"final"`;
  - не имеет `tag_id` и `top_k` — это LLM‑only-шаг, который опирается на результаты предыдущих шагов.

### 6.3 Переменные в промптах

Во всех промптах используются фиксированные переменные (mustache-подобный синтаксис):

Для retrieval‑шага:

- `{{query}}` — текущий пользовательский запрос;
- `{{context}}` — текстовый контекст, извлечённый на этом шаге (по `tag_id` и `top_k`);
- `{{history}}` — короткая история диалога (несколько последних сообщений);
- `{{campaign_name}}` — имя активной кампании (если есть);
- `{{domain_name}}` — имя домена.

Для финального шага:

- `{{query}}`;
- `{{history}}`;
- `{{step_results}}` — агрегированный текст/структура с результатами всех шагов;
- `{{step_results.<step_id>}}` — результат конкретного шага.

В UI pipeline‑builder присутствует блок “Доступные переменные” с описанием каждой переменной.

### 6.4 Поведение шагов

- Если retrieval‑шаг по своему тегу не нашёл ни одного документа:
  - шаг помечается со статусом, аналогичным `skipped_no_docs`;
  - в финальный шаг отправляется структурированная информация о том, что шаг выполнен, но контекст пустой;
  - retrieval **не расширяется** до всего домена.
- Все retrieval‑шаги выполняются по всем enabled‑Vault домена.
- Финальный шаг **не делает retrieval**, а использует результаты шагов и историю чата.

## 7. UI‑правила

### 7.1 Окно управления документами

Должно уметь:

- показывать список файлов в Vault;
- запускать индексацию;
- удалять документы (только БД+LanceDB, без удаления файла в ФС);
- создавать/редактировать/удалять теги;
- назначать теги на документы.

### 7.2 Окно кампании

Должно уметь:

- создавать/редактировать кампанию;
- задавать `system_prompt` кампании;
- создавать/редактировать теги кампании;
- управлять кампанийными пайплайнами.

### 7.3 Окно чата

Должно содержать:

- селектор домена;
- селектор кампании с опцией общего режима;
- список чатов выбранного домена;
- (опционально) управление закреплением конкретного пайплайна.

Переключение домена:

- меняет список чатов;
- меняет список кампаний;
- влияет на retrieval (меняется набор Vault и документов).

Переключение кампании:

- меняет фильтрацию retrieval;
- меняет набор доступных пайплайнов.

## 8. Инварианты

- Любой Chat, Campaign, Tag, Pipeline и Vault *обязаны* принадлежать конкретному домену.
- Campaign имеет `domain_id`, а не `vault_id`.
- Tag имеет `domain_id` и опциональный `campaign_id`; к Vault не привязан.
- Chat не имеет `vault_id`; всегда имеет `domain_id`.
- Retrieval и pipeline‑retrieval используют все enabled‑Vault домена.
- В режиме кампании retrieval никогда не “расширяется” на весь домен, если по тегам кампании ничего не найдено.
- Общий пайплайн не может использоваться в режиме кампании, кампанийный — в общем режиме (определяется по `campaign_id`).
- Удаление документа удаляет запись в БД и чанки в LanceDB, но не файл в ФС.
- `campaign.system_prompt`, если задан, полностью заменяет доменный system prompt; если пуст, используется доменный.
