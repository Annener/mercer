# Mercer — итоговая целевая реализация

Документ описывает целевую реализацию системы на основании согласованной концепции из диалога. Формулировки ниже не пересказывают текущий код; это целевая модель, к которой должна быть приведена реализация. Формат документа — GitHub Flavored Markdown.

## Архитектурные принципы

Система строится вокруг **Domain** как обязательной верхнеуровневой сущности. Любая рабочая сущность, влияющая на поиск, чат, кампании и пайплайны, существует только внутри домена.

**Vault** — это файловое хранилище внутри домена. В одном домене может быть много Vault, и все они рассматриваются как набор источников одного доменного пространства знаний.

Индексация отделена от факта появления файла в файловой системе. Новый файл, помещённый в Vault, не участвует в поиске и RAG до явного запуска индексации пользователем из окна управления документами.

## Сущности

### Domain

Domain — обязательный контейнер для всех основных операций системы. Чаты, кампании, теги, документы, пайплайны и настройки промптов всегда интерпретируются в рамках конкретного домена.

### Vault

Vault принадлежит одному домену и содержит физические файлы в файловой системе, например PDF и Markdown. Vault не определяет логическую область поиска для чата; он лишь даёт один из наборов документов, входящих в общий контекст домена.

В retrieval и pipeline execution участвуют только **enabled-Vault** текущего домена.

### Document

Document — это запись в PostgreSQL о файле, найденном или импортированном в Vault. У документа есть путь к файлу, статус индексации, технические атрибуты и связи с тегами.

Удаление документа из окна управления документами удаляет запись документа из PostgreSQL и связанные чанки из LanceDB, но не удаляет физический файл из файловой системы.

### Tag

Tag существует **вне Vault** и принадлежит домену. Тег может быть общим доменным или принадлежать конкретной кампании; при этом его можно проставлять документам из любого Vault того же домена.

Если тег создаётся из окна кампании, он автоматически получает принадлежность к этой кампании. Если тег создаётся из окна управления файлами, он считается общим доменным тегом и не связан с кампанией.

### Campaign

Campaign принадлежит домену, а не Vault. Кампания использует все Vault домена как потенциальные источники, но фактически сужает доступный контекст по совокупности своих тегов.

Кампания может иметь собственный `system_prompt`, который используется при генерации ответа вместо доменного системного промпта, если активна именно эта кампания. Если `campaign.system_prompt` пустой, используется доменный системный промпт.

### Chat

Chat всегда принадлежит домену. У чата нет привязки к Vault, потому что рабочий контекст чата формируется по всем Vault домена.

У чата может быть выбрана активная кампания или включён общий режим. При выбранной кампании контекст RAG фильтруется только документами, подходящими под теги этой кампании; в общем режиме теги кампании не участвуют в фильтрации контекста.

### Pipeline

Pipeline всегда принадлежит домену. Пайплайн либо общий, либо привязан к конкретной кампании.

Общий пайплайн доступен только в общем режиме чата. Кампанийный пайплайн доступен только при выборе своей кампании.

Структура пайплайна полностью хранится в JSON (`steps` и `final_step`). Дополнительная таблица `pipeline_labels` в целевой модели **не используется**.

## Индексация

Пользователь кладёт файл в Vault на уровне файловой системы. После этого система может видеть файл как кандидат на обработку, но файл ещё не должен участвовать в поиске, пока пользователь явно не запустит индексацию из окна управления документами.

Компонент `rag-indexer` выполняет распознавание, чанкинг, эмбеддинг и запись результата в LanceDB. Метаданные документа и его административное состояние хранятся в PostgreSQL.

Индексация должна быть идемпотентной по документу и поддерживать переиндексацию. При удалении документа из интерфейса удаляются запись документа и его чанки, но не исходный файл.

## Работа тегов

Теги являются доменными и могут назначаться документам из любого Vault внутри этого домена. Это позволяет использовать теги как логическую сквозную систему классификации поверх нескольких Vault.

Поддерживаются два режима создания тегов:
- Из окна кампании — создаётся тег с `campaign_id`, автоматически принадлежащий кампании.
- Из окна управления файлами — создаётся общий тег домена без `campaign_id`.

Тег кампании можно назначать на документы любых Vault домена, если эти документы должны участвовать в контексте данной кампании. Общие теги можно использовать для материалов общего назначения: справок, инструкций мастера, генераторов и других доменных источников.

## Контекст чата

Чат создаётся в конкретном домене и при переключении домена пользователь видит только чаты этого домена. Это означает, что список чатов, список кампаний и набор доступных документов всегда зависят от выбранного домена.

У чата есть селектор кампании. Он допускает два состояния:
- Выбрана конкретная кампания.
- Выбран общий режим.

Если выбрана кампания, то retrieval обязан ограничивать доступные документы только теми, которые размечены тегами этой кампании. Если выбран общий режим, retrieval работает по всем документам домена без кампанийной фильтрации по тегам.

Общий режим нужен для поиска сведений, когда пользователь не помнит, к какой кампании относится источник. Это именно режим без участия кампанийных тегов в фильтрации контекста.

## Retrieval и RAG

Базовый поиск и RAG должны работать на уровне **всех enabled-Vault домена**, а не одного Vault. Это касается и обычного ответа чата, и пайплайнов.

Внутренне retrieval должен принимать `domain_id` и вычислять список активных Vault домена. Поиск по векторному индексу может выполняться как multi-vault retrieval с последующим объединением и дедупликацией результатов.

Фильтрация по документам должна строиться так:
1. Если активна кампания, вычисляются разрешённые теги кампании в рамках домена.
2. По этим тегам находятся документы во всех enabled-Vault домена.
3. Retrieval ограничивается только найденными документами.
4. Если режим общий, фильтрации по кампанийным тегам нет.

Если у кампании есть теги, но ни один документ по ним не проиндексирован, retrieval должен возвращать пустой результат, а не падать обратно на весь домен. Это защищает инвариант фильтрации кампанийного контекста.

## Pipeline

Пайплайн — это механизм улучшенной генерации, работающий поверх того же доменного пространства документов. Он не привязан к Vault и может читать документы из любых enabled-Vault домена.

Пайплайн состоит из шагов. Retrieval-шаги берут контекст по своему `tag_id`, формируют промежуточный результат и затем передают его в финальный шаг. Финальный шаг не делает retrieval и собирает итоговый ответ только на основании результатов предыдущих шагов, истории диалога и текущего пользовательского запроса.

Шаги пайплайна могут выполняться параллельно. Для потоковой выдачи ответа используется SSE через `StreamingResponse` с `media_type="text/event-stream"`, что соответствует типовой схеме серверного однонаправленного стриминга в FastAPI.

### Формат pipeline JSON

Структура пайплайна хранится в JSON и имеет вид:

```json
{
  "version": "2",
  "steps": [
    {
      "id": "rules",
      "type": "retrieval",
      "title": "Правила боя",
      "tag_id": "UUID-тега",
      "top_k": 5,
      "prompt": "..."
    },
    {
      "id": "lore",
      "type": "retrieval",
      "title": "Лор региона",
      "tag_id": "UUID-тега",
      "top_k": 8,
      "prompt": "..."
    }
  ],
  "final_step": {
    "type": "final",
    "prompt": "..."
  }
}
```

Правила формата:
- `steps[]` содержат только retrieval-шаги.
- Каждый retrieval-шаг имеет ровно один `tag_id`.
- У retrieval-шагов обязательны `id`, `title`, `top_k`, `prompt`.
- `id` шага генерируется builder-ом и уникален в рамках пайплайна.
- `final_step` обязателен.
- `final_step` не содержит `tag_id` и `top_k`.
- Поле `mode` внутри pipeline JSON не используется.
- Режим пайплайна определяется только через `pipelines.campaign_id`.

### Доступность пайплайнов

Логика доступности пайплайна определяется режимом чата:
- Если у чата выбран общий режим, доступны только пайплайны домена без `campaign_id`.
- Если у чата выбрана кампания, доступны только пайплайны этой кампании.

Роутер пайплайнов обязан сначала отфильтровать список пайплайнов по режиму, и только потом отдавать их на LLM-выбор. Это не эвристика, а жёсткое бизнес-правило.

Если пользователь вручную заблокировал пайплайн, блокировка действует только пока пайплайн соответствует текущему режиму. При смене кампании или переходе в общий режим несовместимая блокировка должна быть проигнорирована.

### Теги в шагах пайплайна

Если шаг пайплайна указывает `tag_id`, этот тег сначала проверяется на допустимость в рамках текущего режима. Для кампании это означает, что шаг может использовать теги своей кампании и разрешённые общие доменные теги. Для общего режима шаг может использовать только общие доменные теги.

Если шаг после проверки не имеет допустимого тега или не находит ни одного документа, шаг должен давать пустой результат со статусом `skipped_no_docs`. Он не должен расширять поиск до всего домена автоматически.

### Доступные переменные в UI builder

UI pipeline-builder должен показывать блок **"Доступные переменные"** с подсказками по использованию переменных в промптах.

Для retrieval-шагов доступны:
- `{{query}}`
- `{{context}}`
- `{{history}}`
- `{{campaign_name}}`
- `{{domain_name}}`

Для финального шага доступны:
- `{{query}}`
- `{{history}}`
- `{{step_results}}`
- `{{step_results.<step_id>}}`

## UI-правила

### Окно управления документами

Окно управления документами должно поддерживать:
- Запуск индексации по найденным или добавленным файлам.
- Удаление записи документа из БД и его чанков из LanceDB без удаления физического файла.
- Создание, редактирование и удаление тегов.
- Назначение тегов документам.

### Окно управления кампанией

Окно кампании должно поддерживать:
- Создание и редактирование самой кампании.
- Создание тегов кампании.
- Управление пайплайнами кампании.
- Просмотр и изменение `system_prompt` кампании при наличии такого функционала.

### Окно чата

В окне чата должны быть:
- Переключатель домена.
- Переключатель кампании с вариантом общего режима.
- Список доменных чатов, автоматически ограниченный выбранным доменом.
- Возможность ручной блокировки пайплайна, если это предусмотрено UI.

Переключение домена должно менять и набор доступных чатов, и список кампаний, и рабочее пространство retrieval. Переключение кампании должно менять правила фильтрации контекста и список доступных пайплайнов.

## Целевая модель БД

Ниже — целевой состав ключевых полей.

### `domains`

- `domain_id` PK
- `display_name`
- `description`
- `enabled`
- `created_at`
- `updated_at`

### `vaults`

- `vault_id` PK
- `domain_id` FK -> `domains.domain_id`
- `display_name`
- `enabled`
- настройки индексации и эмбеддингов

### `campaigns`

- `id` PK
- `domain_id` FK -> `domains.domain_id`
- `name`
- `description`
- `system_prompt`
- `created_at`
- `updated_at`

### `tags`

- `id` PK
- `domain_id` FK -> `domains.domain_id`
- `campaign_id` FK nullable -> `campaigns.id`
- `name`
- `color`
- `created_at`
- уникальность: `UNIQUE(name, domain_id, campaign_id)`

### `documents`

- `id` PK
- `vault_id` FK -> `vaults.vault_id`
- `source_path`
- `title`
- `status`
- `indexed_at`
- `created_at`
- уникальность: `UNIQUE(vault_id, source_path)`

### `document_labels`

- `document_id` FK -> `documents.id`
- `tag_id` FK -> `tags.id`
- composite PK: `(document_id, tag_id)`

### `pipelines`

- `id` PK
- `pipeline_id`
- `domain_id` FK -> `domains.domain_id`
- `campaign_id` FK nullable -> `campaigns.id`
- `version`
- `name`
- `description`
- `steps` JSONB
- `final_composition` JSONB
- `is_active`
- `created_at`
- уникальность: `UNIQUE(pipeline_id, version)`

Таблица `pipeline_labels` в целевой модели отсутствует.

### `chats`

- `id` PK
- `title`
- `domain_id` FK -> `domains.domain_id`, `NOT NULL`
- `campaign_id` FK nullable -> `campaigns.id`
- `locked_pipeline_id` nullable
- `pipeline_versions` JSONB
- `created_at`
- `updated_at`

`Chat.vault_id` в целевой модели отсутствует, потому что чат работает на уровне всех Vault домена.

### `messages`

- `id` PK
- `chat_id` FK -> `chats.id`
- `role`
- `content`
- `pipeline_id` FK nullable -> `pipelines.id`
- `created_at`

## Целевая логика API

### Создание чата

При создании чата обязателен `domain_id`. `campaign_id` может быть пустым, что означает общий режим.

Чат не должен принимать `vault_id`. Все дальнейшие операции retrieval вычисляют рабочий набор Vault через `domain_id`.

### Отправка сообщения

При отправке сообщения поток обработки должен быть таким:
1. Сохранить пользовательское сообщение.
2. Определить активный домен и кампанию.
3. Передать чат в `pipeline_router`.
4. Если выбран пайплайн — выполнить его и стримить ответ.
5. Если пайплайн не выбран — выполнить обычный multi-vault RAG и сгенерировать ответ.

Артефакт `_run_pipelines` в этой модели не нужен. Гибридный режим "обычный RAG плюс скрытые фоновые пайплайны" не входит в целевую архитектуру.

### Контекст для `pipeline_router` и `pipeline_executor`

В `chat_context` должны передаваться минимум:
- `chat_id`
- `domain_id`
- `campaign_id`
- `vault_ids` всех активных Vault домена
- `history`
- `collected_fields`
- `mode`
- `confidence`
- `reasoning`
- `config`

### `pipeline_router`

`pipeline_router` должен:
1. Получить активные пайплайны домена.
2. Предфильтровать их по `campaign_id` и текущему режиму.
3. Проверить `locked_pipeline_id` на совместимость с режимом.
4. Сформировать промпт с указанием текущего режима.
5. Выбрать подходящий пайплайн или вернуть `None`.

### `pipeline_executor`

`pipeline_executor` должен:
1. Работать по `domain_id` и `vault_ids`, а не по одному `vault_id`.
2. Для каждого шага вычислять разрешённые теги через домен и кампанию.
3. Выполнять retrieval по всем enabled-Vault домена.
4. Передавать в LLM шага не только `system` и текущий `user` query, но и короткую историю диалога.
5. Выполнять retrieval-шаги параллельно, если между ними нет логических зависимостей.
6. Передавать в финальный шаг результаты retrieval-шагов, включая статусы вроде `skipped_no_docs`.
7. Финализировать общий ответ на основании результатов шагов.

## Инварианты

Ниже перечислены правила, которые должны сохраняться в любой реализации.

- Домен обязателен для чата, кампании, Vault и пайплайна.
- Кампания принадлежит домену, а не Vault.
- Тег принадлежит домену, а не Vault.
- Тег кампании может назначаться документам любых Vault своего домена.
- Чат не привязан к Vault.
- Обычный RAG и pipeline RAG работают по всем enabled-Vault домена.
- В режиме кампании retrieval не должен расширяться обратно на весь домен, если по тегам кампании не найдено документов.
- Общий пайплайн не может быть активен в режиме кампании.
- Кампанийный пайплайн не может быть активен в общем режиме.
- Pipeline JSON является единственным источником истины для шагов пайплайна.
- Таблица `pipeline_labels` в целевой модели отсутствует и не используется.
- Удаление записи документа удаляет только БД-запись и чанки индекса, но не файл в файловой системе.

## Минимальный план миграции

1. Перевести `Campaign` с `vault_id` на `domain_id`.
2. Перевести `Tag` с `vault_id` на `domain_id`, сохранив `campaign_id` как nullable.
3. Добавить `campaign_id` в `Pipeline`.
4. Удалить `Chat.vault_id`.
5. Сделать `Chat.domain_id` обязательным (`NOT NULL`).
6. Переписать retrieval и pipeline execution на multi-vault модель.
7. Удалить `_run_pipelines` как мёртвую ветку логики.
8. Удалить `pipeline_labels` и связанный с ней код.
9. Добавить жёсткую фильтрацию пайплайнов по режиму кампании в роутере.

## Что считается корректной реализацией

Реализация считается соответствующей концепции, если пользователь может создать домен, добавить в него несколько Vault, проиндексировать документы, разметить документы сквозными доменными и кампанийными тегами, выбрать кампанию в чате и получить RAG-контекст только по документам, разрешённым этой кампанией, независимо от того, в каком Vault лежат документы.

Реализация также считается корректной, если в общем режиме пользователь ищет по всему домену без участия кампанийных тегов, а пайплайны автоматически ограничиваются своим режимом доступности и не пересекают границы кампаний.
