# Mercer — дополнительный контекст по согласованной целевой реализации

Документ фиксирует дополнительные договорённости, которые должны использоваться вместе с основным концептом и реестром багов. Он описывает целевую схему БД, требования к SSE и параллельному выполнению шагов, состав `chat_context` и прямой запрет на легаси-логику.

## 1. Целевая схема БД

Ниже фиксируются только те элементы схемы, которые были отдельно запрошены как критичные для реализации.

### 1.1 Таблица `messages`

Назначение: хранение истории диалога внутри чата.

Целевой состав полей:

- `id` — PK, UUID.
- `chat_id` — FK -> `chats.id`, `NOT NULL`.
- `role` — строковое поле роли сообщения (`user`, `assistant`, при необходимости системные технические роли).
- `content` — текст сообщения, `NOT NULL`.
- `pipeline_id` — FK nullable -> `pipelines.id`.
- `created_at` — timestamp создания сообщения.

Требования:

- Сообщение всегда принадлежит конкретному чату.
- Сообщение ассистента, сгенерированное через pipeline, может содержать ссылку на `pipeline_id`.
- Сообщение не должно зависеть от `vault_id`, потому что чат работает на уровне домена.

### 1.2 Явное решение по `pipeline_labels`

Таблица `pipeline_labels` **не входит в целевую модель** и должна быть удалена из архитектуры.

Правила:

- `pipeline_labels` не используется ни как источник истины, ни как индекс.
- Структура пайплайна полностью хранится в JSON (`steps` и `final_step`).
- Все retrieval-шаги содержат свой `tag_id` непосредственно внутри JSON.
- Backend, router, executor и UI не должны зависеть от `pipeline_labels`.
- Любая легаси-логика, которая читает или пишет `pipeline_labels`, подлежит удалению.

### 1.3 UNIQUE-констрейнты

Обязательные ограничения:

#### `tags`

```sql
UNIQUE(name, domain_id, campaign_id)
```

Смысл:

- имя тега уникально в рамках домена и кампании;
- одинаковое имя может существовать:
  - в разных доменах;
  - как общий тег и как тег другой кампании, если это сознательно допускается моделью;
- после перехода на доменную модель `vault_id` не участвует в уникальности.

#### `documents`

```sql
UNIQUE(vault_id, source_path)
```

Смысл:

- в одном Vault один и тот же путь к файлу не должен дублироваться;
- одинаковые `source_path` в разных Vault допустимы.

#### `pipelines`

```sql
UNIQUE(pipeline_id, version)
```

Смысл:

- логический идентификатор пайплайна и его версия образуют уникальную пару;
- поддерживается версионирование пайплайнов.

#### `domain_prompts`

```sql
UNIQUE(domain_id, prompt_type)
```

Смысл:

- в домене не должно быть двух промптов одного типа.

#### `domain_clarification_fields`

```sql
UNIQUE(domain_id, field_name)
```

Смысл:

- имя поля уточнения уникально в рамках домена.

## 2. Требование к SSE и параллельному выполнению шагов

### 2.1 SSE

Pipeline execution должен стримить ответ через **Server-Sent Events**.

Требование:

- серверный endpoint пайплайна возвращает `StreamingResponse`;
- `media_type` должен быть:

```python
text/event-stream
```

Назначение SSE:

- однонаправленный поток от сервера к клиенту;
- передача статусов выполнения шагов;
- передача partial/final токенов финального ответа;
- передача диагностических событий (`started`, `step_started`, `step_completed`, `step_skipped_no_docs`, `final_started`, `done`, `error`).

### 2.2 Параллельность шагов

Retrieval-шаги пайплайна должны выполняться параллельно, если между ними нет логической зависимости.

Целевая модель:

- все retrieval-шаги независимы друг от друга;
- каждый retrieval-шаг:
  1. получает свой `tag_id`, `top_k`, `prompt`;
  2. выполняет retrieval по всем enabled-Vault домена;
  3. формирует свой промежуточный результат;
  4. публикует статус в SSE;
- после завершения всех retrieval-шагов запускается **ровно один** финальный шаг;
- финальный шаг всегда выполняется последним и не запускается параллельно с retrieval-шагами.

### 2.3 Поведение при отсутствии документов

Если retrieval-шаг не нашёл документов по своему тегу:

- шаг получает статус `skipped_no_docs`;
- retrieval не расширяется до всего домена;
- шаг не считается ошибкой выполнения;
- информация о пропуске должна быть передана в финальный шаг как часть результатов pipeline;
- в SSE должен уходить отдельный event/status для такого шага.

### 2.4 Ошибки и логирование

Если внутри pipeline execution происходит исключение:

- ошибка должна логироваться с traceback;
- клиент должен получить SSE-событие ошибки;
- executor не должен молча глотать исключения.

## 3. Состав объекта `chat_context`

`chat_context` — это нормализованный объект контекста, который используется роутером и экзекьютором пайплайнов. Он должен быть единым по смыслу и не тащить legacy-поля вроде одиночного `vault_id`.

### 3.1 Обязательный состав `chat_context`

Минимальный состав:

- `chat_id` — идентификатор чата.
- `domain_id` — идентификатор домена.
- `campaign_id` — идентификатор кампании или `null` для общего режима.
- `vault_ids` — список всех enabled-Vault домена.
- `history` — короткая история диалога.
- `collected_fields` — собранные поля/уточнения, если используются в домене.
- `mode` — логическое представление режима чата (`general` или `campaign`) для удобства runtime-кода.
- `confidence` — confidence от pipeline router, если пайплайн был выбран через роутер.
- `reasoning` — reasoning от pipeline router, если он используется в диагностике/UI.
- `config` — runtime-конфиг, необходимый экзекьютору.

### 3.2 Использование в router

`pipeline_router` использует из `chat_context` как минимум:

- `chat_id`;
- `domain_id`;
- `campaign_id`;
- `mode`;
- `history` (если учитывается в prompt выбора);
- `config`.

Router обязан:

1. Получить пайплайны домена.
2. Предфильтровать их по `campaign_id` и текущему режиму.
3. Проверить совместимость `locked_pipeline_id` с режимом.
4. Передать в LLM только валидные для текущего режима варианты.

### 3.3 Использование в executor

`pipeline_executor` использует из `chat_context` как минимум:

- `chat_id`;
- `domain_id`;
- `campaign_id`;
- `vault_ids`;
- `history`;
- `collected_fields`;
- `mode`;
- `config`.

Executor обязан:

- работать по `domain_id` и `vault_ids`, а не по одному `vault_id`;
- вычислять retrieval-контекст шага с учётом режима и кампании;
- передавать историю диалога в LLM-вызовы шагов и финального шага;
- формировать структурированные результаты шагов для финализации.

## 4. Явный запрет на легаси-логику

### 4.1 Удаление `_run_pipelines`

Функция `_run_pipelines` считается legacy-веткой и должна быть удалена из `chat.py`.

Причина:

- она относится к старой гибридной идее “обычный RAG + фоновые пайплайны”;
- целевая архитектура Mercer больше не использует эту модель;
- наличие `_run_pipelines` создаёт ложное впечатление существования второго способа enrichment.

### 4.2 Запрет на гибридные режимы

В целевой архитектуре **запрещена** следующая логика:

- сначала обычный RAG-ответ;
- затем скрытые или фоновые pipeline enrichment-шаги;
- затем смешивание этих результатов в одном неявном flow.

Допустимы только два сценария обработки сообщения:

1. **Router выбрал pipeline** -> выполняется pipeline execution -> ответ полностью строится через pipeline.
2. **Router не выбрал pipeline** -> выполняется обычный multi-vault RAG -> ответ строится без pipeline.

Смешанный режим не допускается.

### 4.3 Запрет на legacy `vault_id`-ориентированный flow

Запрещены следующие legacy-паттерны:

- использование `Chat.vault_id` как источника retrieval-контекста;
- выбор “первого Vault домена” для pipeline execution;
- вызов pipeline executor с одиночным `vault_id` вместо `domain_id + vault_ids`;
- логика, в которой чат считается привязанным к одному Vault.

Целевая модель всегда доменно-ориентирована.

### 4.4 Запрет на fallback до всего домена при пустом кампанийном контексте

Если чат работает в режиме кампании и retrieval по тегам кампании не дал документов:

- retrieval возвращает пустой результат;
- pipeline шаг получает `skipped_no_docs`;
- система не должна автоматически откатываться к поиску по всему домену.

Это обязательный инвариант.

### 4.5 Запрет на использование `pipeline_labels`

Так как `pipeline_labels` исключена из целевой модели, запрещается:

- читать `pipeline_labels` в router;
- читать `pipeline_labels` в executor;
- писать `pipeline_labels` при сохранении пайплайна;
- использовать `pipeline_labels` в builder/UI;
- строить валидацию пайплайна через `pipeline_labels`.

Все правила пайплайна должны читаться только из:

- `pipelines.campaign_id`;
- JSON-структуры пайплайна (`steps`, `final_step`).
