# Vault Watchdog — концепт

## Проблема

Индексация в системе запускается только вручную через API или UI. Если пользователь
добавил файл в vault, отредактировал или удалил существующий — система об этом не
знает до следующего ручного запуска. Нужен механизм автоматического обнаружения
изменений и управляемой реакции на них.

## Что мы строим

**Vault Watchdog** — фоновый asyncio-task внутри `rag-indexer`, который каждые 60
секунд сканирует все активные vault-директории, сравнивает содержимое с Redis-кэшем
и принимает решение: запустить авто-индексацию, пометить файл как ожидающий ручной
обработки, или удалить исчезнувший документ из всех хранилищ.

Поведение watchdog-а определяется глобальной настройкой в PostgreSQL, которую
пользователь меняет через UI. Фронтенд (внутри `rag-backend`) показывает баннер в
окне чата если есть файлы, ожидающие ручной индексации.

## Три сценария поведения

Вся логика сводится к одному параметру в `platform_settings`:
`watchdog_auto_index_extensions` — список расширений для авто-индексации.

| Сценарий | Значение параметра | Поведение |
|---|---|---|
| 1 — всё автоматически | `.md,.pdf` | Любое изменение → watchdog запускает задачу |
| 2 — выборочная авто-индексация | `.md` | `.md` — авто, `.pdf` — помечается как pending, задача не запускается |
| 3 — только ручная | `` (пусто) | Watchdog следит и помечает, задач не запускает никогда |

В сценариях 2 и 3 в окне чата появляется баннер с количеством ожидающих файлов и
кнопкой ручного запуска индексации.

Дефолтное значение при первом запуске: `.md,.pdf` (Сценарий 1).

## Как watchdog обнаруживает изменения

Каждые 60 секунд для каждого активного vault:

1. **Скан диска** через существующий `scan_vault()`. Оптимизация: сначала сравнивается
   `st_mtime` из Redis-кэша — если время модификации не изменилось, md5 не
   пересчитывается. md5 считается только для файлов с изменившимся `mtime`.

2. **Сравнение с Redis-кэшем** `vault:{vault_id}:files`, который хранит `indexed_md5`
   и `mtime` для каждого файла.

3. **Классификация изменений:**
   - *Новый* — есть на диске, нет в Redis
   - *Изменён* — `mtime` отличается и md5 не совпадает с `indexed_md5`
   - *Удалён* — есть в Redis, нет на диске

4. **Реакция** согласно настройке расширений и наличию активной задачи для vault.

## Логика реакции

```
для каждого vault:
  diff = [новые] + [изменённые]
  удалённые → немедленно удалить из LanceDB + PostgreSQL + Redis

  to_auto = [f for f in diff if f.ext in auto_extensions]
  to_mark = [f for f in diff if f.ext not in auto_extensions]

  for f in to_mark:
      state_manager.mark_file_pending(vault_id, f.path)

  if to_auto and vault_id not in active_tasks:
      indexer_service.start_task(vault_id)
```

## Удаление файлов

Если файл исчез с диска — watchdog выполняет атомарную последовательность:

1. Удалить чанки из **LanceDB** по `document_id` (новый метод `delete_chunks_by_document_id`)
2. Удалить запись из **PostgreSQL** таблицы `documents` (новый метод `delete_document`)
3. Удалить ключ из **Redis**-кэша vault (новый метод `remove_file_from_vault_cache`)

Это единственный случай когда watchdog сам модифицирует данные в хранилищах, не
делегируя это `indexer_service`.

## Настройка и перечитывание

Параметр `watchdog_auto_index_extensions` хранится в уже существующей таблице
`platform_settings` (тип `str`, значение — строка вида `.md,.pdf`).

Watchdog читает параметр из БД **на каждой итерации** — один `SELECT` раз в 60
секунд. Изменение настройки в UI применяется на следующей итерации без перезапуска
сервиса. Pub/sub и reload-endpoint не требуются.

## Что появляется в UI

### Настройки платформы (вкладка «Параметры»)

Секция «Индексация» — список чекбоксов с доступными расширениями (`.md`, `.pdf`).
Выбранные расширения записываются в `platform_settings` через rag-backend API.

### Баннер в окне чата

Показывается если хоть в одном vault есть файлы со статусом `pending`. Содержит:
- Информационный текст с количеством ожидающих файлов
- Кнопку «Запустить индексацию» → `POST /api/v1/tasks` для соответствующего vault

Фронтенд опрашивает endpoint `GET /api/v1/vaults/pending-files` раз в 30 секунд.
Баннер скрывается когда `has_pending: false`.

## Граница ответственности компонентов

| Компонент | Что делает |
|---|---|
| **rag-indexer** | Watchdog loop, scan+diff, delete из LanceDB+PG+Redis, start_task |
| **rag-backend** | API настроек watchdog, endpoint pending-files, фронтенд (UI настроек + баннер) |
| **PostgreSQL** | Хранит `watchdog_auto_index_extensions` в `platform_settings`, documents registry |
| **Redis** | Vault-кэш как источник истины для diff (`indexed_md5`, `mtime`) |

## Что не затрагиваем

- Логика индексации (`indexer_worker`, `indexer_service`) не меняется — watchdog
  вызывает существующий `start_task(vault_id)`
- Redis-state и TTL-стратегия — не меняются
- Настройка per-vault — не реализуется, настройка глобальная
- Интервал watchdog вынесен в env-переменную `WATCHDOG_INTERVAL_SEC` (default: `60`)
