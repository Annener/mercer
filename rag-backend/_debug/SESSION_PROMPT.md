# Стандартный промт для сессии исправления багов

Копируй этот промт в начале каждой сессии, подставив нужный ID эндпоинта.

---

```
Репозиторий: https://github.com/Annener/mercer
Проект: Mercer — RAG-чат, FastAPI бэк + JS фронт

## Контекст (читать перед работой)
- Архитектура и инварианты: concept_plan/arch.md
- Дополнительный контекст: concept_plan/another_context.md
- API контракты: rag-backend/_debug/CONTRACTS.md
- Лог багов: rag-backend/_debug/AUDIT.md
- Прогресс фронт-фиксов: rag-backend/_debug/PROGRESS.md

## Правила
1. Источник правды: concept_plan/arch.md + бэкенд-код. Фронт правится под бэк.
2. Проверка идёт строго по слоям:
   arch.md → models.py → pydantic-схемы → роут → сервисы → CONTRACTS.md → фронт (api.js)
3. За сессию — один эндпоинт, полная цепочка сверху вниз.
4. Сначала аудит → фиксируем все баги в AUDIT.md → только потом правки.
5. После каждого исправления — обновляем статус в AUDIT.md (🔴/⚠️ → ✅).
6. Один баг = один коммит с ID в сообщении (например: `fix: A02 — add pipeline_versions to Chat`).
7. Фронт-файл читаем и правим только через GitHub MCP.

## Цепочка проверки (шаблон)
[ ] arch.md — инвариант для этой сущности
[ ] models.py — поля, типы, nullable, отсутствующие поля
[ ] pydantic-схемы — что принимает и возвращает роут
[ ] роут — бизнес-логика, helper-функции, обращения к полям модели
[ ] сервисы — если роут вызывает executor/router/service
[ ] CONTRACTS.md — финальный контракт
[ ] фронт (api.js / нужный JS-файл) — соответствие контракту

## Текущая задача
Эндпоинт: [ID из CONTRACTS.md, например C1]
Действие: [audit — только аудит | fix — только правки | both — аудит + правки]

Начни с: прочитай AUDIT.md, затем arch.md (раздел нужной сущности),
затем пройди по цепочке проверки.
```
